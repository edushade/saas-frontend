faq:
  title: "Role Permissions FAQ"
  description: "How permissions, roles, overrides and the RBAC engine work inside Edushade."
  items:
    - question: "How many permissions ship out of the box?"
      answer: "295 permissions, organised into 55 functional groups — one group per feature area. The list lives in the auth-service's permissions.json and is loaded by a seeder on a fresh install. As Edushade ships new features, new permission groups land in the same file, so the catalog grows with the platform."
    - question: "What does a permission look like?"
      answer: "A row on the permissions table with a name (`course.view`), a label (`View Course`), a description, a group (`course`) and a guard (`web` by default). The middleware guarding a route checks the name; the admin permission picker reads from the label, description and group. Both layers stay aligned because they share the same source."
    - question: "How do I create a role?"
      answer: "Open Admin → Platform → People → Roles & Permissions and use the Create Role flow, or call POST /rbac/roles. A role carries a name, a label, a description, a status (active / inactive), a guard, plus a permission set you tick on the same form. Save — the role is live and assignable to any user in the tenant."
    - question: "Can I clone an existing role?"
      answer: "Yes. POST /rbac/roles/:roleID/clone duplicates the source role with the same permission set. Useful for templating a TA role from an instructor, a senior moderator from a moderator, or a billing operator from an admin. Adjust the permissions on the clone and you have a new role without re-ticking 50 boxes."
    - question: "What roles ship as bootstrap defaults?"
      answer: "Six seeds: super_admin (no explicit permissions — special case), admin (broad permission set across user, role, tenant, file, notification, analytics, settings), user_manager (user + role assignment scope), developer (tenant + user + file + notification + analytics scope), moderator (user list + sessions + file ops + notifications + analytics view) and auditor (analytics view + export). Real role design happens per tenant on top of these — clone or build new."
    - question: "How are roles assigned to users?"
      answer: "Through the user_roles pivot. POST /rbac/users/:userID/roles/:roleID assigns; DELETE removes. A user can hold multiple roles — the access surface is the union. Invitations carry role_ids[] so a new user lands with the right roles already assigned the moment they accept the invitation."
    - question: "What is a per-user permission override?"
      answer: "A row on user_permissions that grants or revokes a single permission to a single user, independent of any role. Useful for one-off elevations (a moderator with delete access for a specific incident) or one-off reductions (an instructor who should not see analytics for one quarter). The middleware merges role-based and user-level grants when deciding access."
    - question: "How does permission resolution actually work?"
      answer: "On every protected request, middlewares.Can(name, pp) asks the PermissionProvider whether the user has the permission. The provider looks up the user's roles, gathers their role_permissions, adds the user_permissions overrides, and answers yes/no. The result is cached per request to keep the lookup cheap, and the same logic powers the admin sidebar's render decisions."
    - question: "Where is RBAC enforced?"
      answer: "Two places. Server-side: every protected route across every microservice is wrapped in middlewares.Can(permission_name, pp). Client-side: the admin sidebar config carries requireRoles + requirePermissions per item, and items hide when the user lacks them. The two share the same permission names, so a feature gated server-side is also gated in the menu — no drift, no greyed-out items."
    - question: "Are RBAC changes audited?"
      answer: "Yes. Role create / update / delete, permission grants and revocations, user-role assignments and impersonation events all flow through the audit-service. Admin → Platform → Audit → Audit Logs surfaces them with the actor, the target, the change and the timestamp. Activity Logs covers the broader auth event stream."
    - question: "Is RBAC tenant-isolated?"
      answer: "Hard-isolated. Every roles row, role_permissions row, user_roles row and user_permissions row carries a tenant_id. Queries always filter by it; the API never returns roles or permission assignments from another tenant. The 295 base permissions are global definitions, but their assignment to roles and users is fully per-tenant."
    - question: "Can RBAC integrate with SSO or directory groups?"
      answer: "Yes. The OAuth-linked event is tracked on the user record; tenant invitations carry role_ids[] and permission_ids[] for first-touch assignment; the system-token endpoints expose role and permission management for inter-service use. Combine those with an SSO claim mapper and a directory sync, and group-based assignments fold into the same RBAC engine."
    - question: "How does the admin sidebar know what to hide?"
      answer: "The sidebar config in `components/admin/partials/sidebar-navigation.ts` tags each item with `requireRoles` and `requirePermissions`. The render function checks the signed-in user's roles and permissions and skips items that fail. Custom roles automatically get a custom menu — a new role created in Roles & Permissions sees exactly the items you grant it, no code change required."
    - question: "Can I disable a role without losing its assignments?"
      answer: "Yes. Set the role status to `inactive`. The role still exists, the user_roles assignments still exist, but the middleware treats inactive roles as not granting any permission. Flip it back to `active` and access resumes — useful for a temporary hold during a re-org or a security review."
    - question: "Are there web-only and API-only permissions?"
      answer: "The guard column on roles and permissions distinguishes contexts (web by default). The same permission name can carry separate web and API guards if your tenant needs the distinction — for example, allowing a role to edit through the admin UI but not through a system-token API path. Most tenants run web-only and let the system token handle inter-service traffic."
