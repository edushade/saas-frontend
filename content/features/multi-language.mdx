faq:
  title: "Multi-Language FAQ"
  description: "How locales, URLs, RTL, fallbacks, typography and region-aware formatting actually work inside Edushade."
  items:
    - question: "Which languages ship by default?"
      answer: "Twenty are ready to enable from the dropdown — English, Spanish, French, German, Italian, Portuguese, Russian, Japanese, Korean, Chinese, Arabic, Hindi, Bengali, Turkish, Dutch, Polish, Swedish, Danish, Norwegian and Finnish — each with the matching flag emoji. The locale code is two characters (`en`, `es`), so a custom locale you add later still slots in."
    - question: "Can I add a language that is not in the preset list?"
      answer: "Yes. The locale schema accepts any code from 2 to 5 characters, a label and a flag, so a locale Edushade does not pre-register can be added by editing the localization config — the rest of the platform treats it the same as a preset locale, including fallback, RTL toggle and per-locale typography."
    - question: "How does right-to-left support work?"
      answer: "Arabic, Hebrew, Persian and Urdu auto-detect as RTL when added. The platform sets the html dir and lang attributes from the active locale, the theme system flips alignment, padding and margin utilities, and the locale switcher itself respects direction. RTL can also be toggled per locale on the Enabled Languages table — useful for an LTR locale that needs to render mixed content."
    - question: "What is a fallback locale?"
      answer: "Each non-primary locale picks one. If a learner is on the Bengali version of a page and the meta title for Bengali is missing, the platform pulls from that locale's fallback (English by default). The chain prevents empty headers, empty meta tags and empty form labels while the team is mid-translation. Primary locale falls back to itself."
    - question: "Which URL structures are supported?"
      answer: "Three: subdirectory (example.com/en/page), subdomain (en.example.com/page) and query parameter (example.com/page?lang=en). Subdirectory is the most SEO-friendly and ships as the default. Subdomain works when each locale has its own host. Query parameter is the lightest to set up if you cannot touch DNS or routing today."
    - question: "How does the platform decide which locale to load?"
      answer: "Resolution runs in this order: explicit URL (path or query), then the Accept-Language header (when Auto-detect User Locale is on), then localStorage (a returning visitor's last choice), then the primary locale. Every step validates against the enabled-locales list, so a stale URL or a header for a disabled language quietly falls back instead of erroring."
    - question: "What does Auto-detect User Locale do?"
      answer: "It enables the Accept-Language fallback in the resolution chain. With it on, a first-time visitor whose browser is set to Spanish lands on the Spanish version of your site even without a `/es` segment in the URL. The locale switcher overrides the auto-detect, and the override is stored in localStorage so it persists between visits."
    - question: "Which content types carry localized values?"
      answer: "Pages (title, slug, meta title, meta description), posts (title, excerpt, content), forms (title, description, every field's label and placeholder, submit-button text, success message), menus (item labels), blocks (content for hero, text and similar text-bearing blocks), SEO (meta title template, meta description, meta keywords) and system-page metadata. Every one uses the same `LocalizedField` shape — switch the locale and the value swaps."
    - question: "Can I use a different font for non-Latin scripts?"
      answer: "Yes. Pick a font-sans, font-serif and font-mono per locale on the design theme. Edushade stores the per-locale typography map and emits :lang() CSS rules — so a learner whose document.lang is `ar` sees the Arabic typeface automatically, while an English learner on the same page keeps yours. Light and dark themes share the typography stack, so the override only needs to be set once per locale."
    - question: "How are dates, times, numbers and currencies formatted?"
      answer: "Localization carries default date format (DD/MM/YYYY, MM/DD/YYYY, YYYY-MM-DD, DD MMM YYYY, MMMM DD, YYYY), time format (12h or 24h), number format (en-US, en-GB, de-DE, fr-FR, es-ES) and a default currency (USD, EUR, GBP, JPY, INR, AUD, CAD) plus timezone. Formatters across the platform read from this config so a single change updates every page that renders a date or a price."
    - question: "Can a learner switch language on the public site?"
      answer: "Yes. The locale switcher renders next to the navigation. Picking a language updates the URL (path, subdomain or query parameter, depending on your URL structure), writes to localStorage so the choice persists and triggers a re-render with the new locale's content. The html lang attribute updates so :lang() CSS rules — including per-locale fonts — switch with it."
    - question: "Does multi-language work with server-side rendering?"
      answer: "Yes. The locale resolver runs both on the server (using the request URL and Accept-Language header) and on the client (using localStorage and the URL). The server picks the right locale before rendering, the html lang attribute is correct on first paint, and the client takes over without reflowing the page."
    - question: "Does the menu and navigation get localized too?"
      answer: "Yes. Menu items carry localized labels, so a Hindi learner sees the menu in Hindi. The CMS surfaces a locale switcher inline on the menu editor, the same way pages and posts do, so a translator can fill in labels without leaving the menu builder."
    - question: "How are SEO fields localized?"
      answer: "Site-wide SEO carries a localized meta title template (with `{page_title} | {site_name}` interpolation), localized meta description and localized meta keywords. Per-page SEO inherits from these defaults and lets you override per locale — so the Spanish version of a page gets a Spanish meta title, a Spanish description and Spanish keywords, while sharing the same template structure."
