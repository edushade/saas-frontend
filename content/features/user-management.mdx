faq:
  title: "User Management FAQ"
  description: "How users are stored, verified, sessioned, invited, exported and supported across the platform."
  items:
    - question: "What does Edushade store on a user record?"
      answer: "Account fields (email, phone, username, password, name, avatar), profile fields (bio, professional summary, designation, expertise, gender, date of birth, address), geo fields (continent, country, city, state, timezone, locale, auto-detected from session), three type flags (is_admin, is_instructor, is_learner), six social links (LinkedIn, Twitter, Facebook, Instagram, YouTube, website) and OAuth provider IDs for Google, Facebook, Apple and GitHub. Everything is tenant-scoped and exportable to CSV."
    - question: "What is the user lifecycle?"
      answer: "Five states: pending_verification → active → inactive / suspended / deleted. New users start in pending_verification until they confirm their email or phone; active is the operating state; inactive and suspended are reversible holds; deleted is a soft delete that preserves the audit trail. Reactivation is a single status change, no data loss."
    - question: "How does verification work?"
      answer: "Email verification via a token-based link or via OTP delivered by email or SMS. Phone verification via OTP. Dedicated change flows for email and phone — when a user changes either, the new value goes through its own verification before it replaces the old one. Verification carries a status (pending, verified, expired, failed) with configurable expiry and a default max-attempts of three."
    - question: "How are sessions handled?"
      answer: "JWT access tokens with a JTI, plus refresh tokens that rotate per device. Each session is its own row carrying device type, brand, model, OS, browser, architecture, IP geo (continent, country, city, region, latitude, longitude, ISP, ASN), timezone and language. The admin can list every active session for a user and revoke a single session or all sessions in one call. A new-device login alert notification fires automatically on the first session from an unfamiliar device."
    - question: "Which OAuth providers are supported?"
      answer: "Four: Google, Facebook, Apple and GitHub. The admin can link or unlink any provider from the user detail page (POST /users/:user_id/oauth/link, /unlink), and OAuth login on the public side links to the same record. The OAuth-linked event is tracked on the notification system so the user gets an in-app, email or push alert when a provider is added."
    - question: "How do invitations work?"
      answer: "An admin opens the invitation flow, picks an email, optionally writes a message, and selects role_ids and permission_ids to preassign. The platform stores a row on the invitations table (organization_id, tenant_id, email, token, role_ids, permission_ids, invited_by, expires_at, status), emails the invitee a token-bearing link, and tracks accept / expire / cancel as explicit statuses. On acceptance, the user record is created and the assignments are made in one transaction."
    - question: "Can I import users in bulk?"
      answer: "Yes. The admin People views ship CSV export by default (every field on the user record), so bulk roundtrips are straightforward. Bulk delete is a one-call action gated by user.delete. The same audit pipeline records the bulk operation, so the admin trail does not lose individual records."
    - question: "What user analytics are available?"
      answer: "Five admin endpoints under /users: /stats for headline counts, /filter-values for distinct values across filterable fields, /field-values for per-field distributions, /analytics/growths for time-series growth and /analytics/overview for a quick-stats card. Permissions are gated through user.view; the admin dashboard reads from the same endpoints."
    - question: "What is the unique ID on a user record?"
      answer: "A per-tenant identifier, generated automatically when a user is inserted. The generator runs on the BeforeAppendModel hook with awareness of is_admin / is_instructor / is_learner, so the prefix encodes the user's primary type. The unique ID is editable from the admin and exposed on the CSV export — useful for cross-system reconciliation when you have an existing identity scheme."
    - question: "How does impersonation work?"
      answer: "POST /auth/impersonate/:user_id, gated by the admin.users permission. The admin gets a token that signs them in as the target user, with the impersonation context preserved server-side so the original admin can be restored. POST /auth/impersonate/stop clears it. The audit trail records every impersonation start and stop, so support actions on a user account are always reviewable."
    - question: "Are users tenant-isolated?"
      answer: "Yes. Every users row carries tenant_id, every query filters by it, and there is no cross-tenant view in the user-facing admin. If the same person belongs to two organizations on Edushade, they hold two user records (one per tenant) with two unique IDs and two sets of role and permission assignments. System-token endpoints exist for inter-service operations, not user-facing cross-tenant access."
    - question: "Can a user have direct permissions on top of their role?"
      answer: "Yes. The user_permissions pivot table on the auth-service stores a user-level grant that sits on top of role-based permissions. The admin can grant or revoke a single permission to a single user without changing their role — useful for one-off elevations (a moderator with delete access for a specific incident) or one-off reductions. The platform's RBAC engine checks both layers when resolving access."
    - question: "Are profile fields localized?"
      answer: "User-controlled fields like bio and professional summary are free-text, but the platform stores the user's preferred locale (auto-detected from session, overridable in the profile) and uses it for every notification, email and rendered label. Geo fields auto-populate from the IP geo lookup on each new session, so timezone, country and city are accurate without the user filling them in."
    - question: "What audit data does the platform record?"
      answer: "created_at, updated_at and deleted_at on every user — soft delete preserves history. Last-login-at and last-login-IP on the user record. Every session lifetime tracked (created, last refresh, revoked). The audit-service captures privileged changes (role assignments, permission grants, status changes) with the actor, the target and the before/after, exposed under Admin → Platform → Audit → Audit Logs."
    - question: "Can self-registration be turned off for some tenants?"
      answer: "Yes. Tenant-level configuration covers signup behavior, and an admin can run an invitation-only tenant where every user lands through the invitation flow. The combination of tenant settings, role-based access and the invitations table covers everything from open self-signup to lockstep invite-only without changing the data model."
