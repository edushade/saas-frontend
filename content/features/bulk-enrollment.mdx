faq:
  title: "Bulk Enrollment FAQ"
  description: "Questions people usually ask before they run their first batch."
  items:
    - question: "How many learners can I enroll at once?"
      answer: "Up to a thousand in a single batch. If you have a bigger list, run it as two or three batches — the workflow is the same, and the result reports are independent so you can keep clean records of each."
    - question: "Which programs can I bulk-enroll into?"
      answer: "Three: a course, a learning path or a specific cohort. Whichever one you opened the Bulk Enroll flow from is what the batch enrolls into — you're never surprised by where people landed."
    - question: "What if some of the learners are already enrolled?"
      answer: "They'll be skipped with a clear reason on the result report — usually 'already enrolled in this program'. The rest of the batch still goes through. You don't have to clean the list before running the action."
    - question: "Can a person be enrolled in two cohorts of the same course at once?"
      answer: "No. The platform checks for that and skips the duplicate with a reason. If someone needs to move from one cohort to another, that's a transfer, not a second enrollment — and the bulk update flow handles it cleanly."
    - question: "Do learners still get the welcome email when they're enrolled in bulk?"
      answer: "Yes. Every learner in the batch gets the same enrollment-created notification a one-by-one enrollment would have triggered — email, in-app, SMS and push, in the language they're set to. Bulk doesn't mean silent."
    - question: "Does the instructor know that a batch came in?"
      answer: "Yes. The instructor on the program gets the standard new-enrollments notification. They also see a bulk-enrollment-created event in their notification feed, so they know it was a group, not one learner trickling in."
    - question: "Can I set a single expiry date for the whole group?"
      answer: "Yes — it's a single field on the batch form. Add a date and every learner in the batch gets that expiry. Leave it blank and the access stays open. If you need different expiries per learner, the standard per-enrollment flow handles that."
    - question: "What happens to the people that get skipped?"
      answer: "Nothing — they stay where they were. The result report tells you why each one didn't make it in (already enrolled, already in another cohort, account inactive, and so on), so you can fix what's fixable and re-run the batch. There's no half-state to clean up."
    - question: "What about updating or removing enrollments in bulk?"
      answer: "Same toolkit. Bulk update lets you change status, extend access or set a new expiry across many enrollments at once. Bulk delete cleans things up when a program retires. Bulk export gives you the full enrollment table as a CSV in one click — useful for audits or external systems."
    - question: "Is there an audit trail?"
      answer: "Yes. Every bulk operation lands as an audit entry — who ran it, against which program, how many landed, how many were skipped, and the timestamp. The same audit log surfaces every other privileged change in the platform, so it's all in one place."
    - question: "What permission does a team member need to run a bulk enroll?"
      answer: "The same one they'd need to enroll a single learner. If a role can enroll one person into a course, it can run a batch into that course — no separate bulk-only permission to manage."
    - question: "Can I do this for scholarships too?"
      answer: "Yes. Scholarships have their own bulk eligibility check — paste in or select the applicants you want to evaluate, and the platform runs the eligibility rules across every one of them at once and gives you a result list. The flow is built for the same scale."
    - question: "Does this work in a private course?"
      answer: "If the course is set to private, the platform protects it — bulk enroll on a private course is blocked at the program level, the same way single enrollment is. Move the course to published or open enrollment first, or invite each learner explicitly."
    - question: "What if the program is full?"
      answer: "If the batch would push the program past its max enrollments cap, the action is rejected before any enrollment is created. You'll see a clear message about the seat limit, and nothing changes on the platform. Trim the list or raise the cap and try again."
