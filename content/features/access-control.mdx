faq:
  title: "Access Control FAQ"
  description: "How access rules, course-level gates, enrollment status and the audit trail come together inside Edushade."
  items:
    - question: "What does Access Control cover in Edushade?"
      answer: "Three layers wired together. Course-level gates (enrollment_open, password_protected, max_enrollments, status) handle the common cases. Per-entity access rules — one row on the access_rules table per entity, with a JSONB filters tree — handle the fine-grained ones. Enrollment status and drip rules handle the time and progression dimension. Together they cover everything from a public freebie to a private invite-only cohort without a code change."
    - question: "Which entity types can carry an access rule?"
      answer: "Nine, in two tiers. Program-level: course, learning_path, cohort, event, bundle. Curriculum-item-level: section, lesson, assignment, quiz, file. The same model and the same admin UI cover all of them, so an access rule on a single lesson behaves the same way as one on a whole course."
    - question: "How rich is the rule builder?"
      answer: "Very. The condition builder pulls from a 69-attribute catalogue across 13 categories: User Profile, Location, Verification, User Type, OAuth, Activity, Enrollment, Completion, Learning Progress, Quiz, Purchase, Engagement and Device. Operators vary by field type — eq / neq / contains / starts_with / ends_with / regex / in for strings, gt / gte / lt / lte / between for numbers, before / after / in_last for dates, geo_within_radius / within_polygon / outside_polygon for location, event_count_in_window / occurred_in_last for behaviour, plus array, boolean, enum and uuid operators. Time units span minutes through years."
    - question: "Can rules use AND/OR logic?"
      answer: "Yes. The filters JSONB shape supports nested AND/OR groups with no depth limit, so a rule can read like (Country in [India, Bangladesh] AND Email Verified = true) OR (Last Login in last 14 days AND Completed Course X). The frontend renders the tree with a builder; the backend stores it verbatim and evaluates against the user catalogue at access time."
    - question: "What happens for anonymous visitors?"
      answer: "Anonymous users (no signed-in user, userID == uuid.Nil) only see entities without an enabled access rule. A program with a disabled rule, an empty filter or no rule at all is accessible to anyone. A program with an enabled, non-empty rule is invisible until they sign in and match. This makes a private course genuinely private — no leakage from the public listing."
    - question: "What if an access rule errors during evaluation?"
      answer: "The platform fails open. If the rule evaluator hits an error — bad JSON, missing field, broken operator — the warning is logged and access is granted. The reasoning: a buggy rule should never accidentally lock out the audience that paid for the program. Combine that with the audit log on every save and the preview API before you ship, and the cost of a failed evaluation stays low."
    - question: "What course-level gates exist alongside access rules?"
      answer: "Four direct flags on every course: enrollment_open (open / closed without un-publishing), password_protected + password (a shared secret on the course page), max_enrollments (a hard seat cap with a live remaining_seats counter), and status (draft, published, archived, private). Cohorts add their own start/end dates and a max-seats cap. Most programs only need these — access rules are the fine-grained tier on top."
    - question: "How does enrollment status affect access?"
      answer: "Five statuses on every enrollment: active, completed, cancelled, expired, suspended. Active and completed grant access; the others hide content. Time-limited enrollments carry an expiry_date and an allow_extensions flag, so a 90-day program either expires automatically or stretches with admin approval. The platform fires expiring-soon and expired notifications so a learner is not surprised by a sudden lock."
    - question: "Can I preview a rule before saving?"
      answer: "Yes. The preview endpoint runs the filter against the live user base and returns three things: total_users (the full audience), estimated_size (how many match the rule) and sample_users (a small sample of matched users). The admin UI surfaces all three so you can confirm the rule lands on the right people before flipping the enabled toggle."
    - question: "Are access rule changes audited?"
      answer: "Yes. Save, delete, enable and disable each fire an audit event with the actor, the resource type (`access_rule`), the resource ID, the entity (entity_type + entity_id), the IP address, the user agent and a timestamp. The audit-service stores them and Admin → Platform → Audit → Audit Logs surfaces them. Every privileged change to access has a paper trail."
    - question: "How is access rule evaluation performant?"
      answer: "The filter service uses an optimised path with a SQL query builder and a per-user cache when both are available — the first match for a user warms the cache, subsequent checks are cheap. The fallback path fetches rules without user matching for surfaces that only need the rule shape. Public listings of programs filter at the entity level: rules block entities, not entire pages, so listing performance scales with the visible set."
    - question: "Are rules tenant-isolated?"
      answer: "Hard-isolated. Every access_rules row carries a tenant_id; every query filters by it; no admin ever sees another tenant's rules. The user catalogue used for matching is itself tenant-scoped, so a rule against `Country = India` evaluates only against this tenant's users, never anyone else's."
    - question: "Who on my team can manage access rules?"
      answer: "Two permissions cover the surface. access_rule.view lets a role read the rule and the field catalogue. access_rule.manage lets a role create, update, delete and toggle rules. Combine those with course.update / cohort.update / learning_path.update for the rest of the program management surface, and an instructor can manage their own programs' access without touching anyone else's."
    - question: "Does Access Control work with subscriptions and bundles?"
      answer: "Yes. Bundles carry their own access rules just like courses. Subscription plans gate content separately through the billing service's content-access selector — a plan can include or exclude specific programs. The two layers compose: a learner needs to clear billing first (purchase, subscription, scholarship or bundle) and then their user attributes need to match any access rule on the entity. Both layers fail closed; together they form a real access surface."
