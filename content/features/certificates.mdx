faq:
  title: "Certificates FAQ"
  description: "How Edushade issues, verifies, expires and revokes certificates across courses, learning paths, events and quizzes."
  items:
    - question: "What is a certificate in Edushade?"
      answer: "A record on the certificates table that says a specific learner finished a specific entity (course, learning path, event or quiz) on a specific date, with a specific grade. It carries a unique number, a verification code and a snapshot of the completion data — not a live join. Once issued, future edits to the source program do not change the certificate."
    - question: "Which programs can issue certificates?"
      answer: "Four entity types are supported today: courses, learning paths, events and quizzes. The same Certificate model handles all four, with entity_type and entity_id pointing at the source record. Cohort-issued certificates additionally store the cohort ID for traceability."
    - question: "When does a certificate get issued automatically?"
      answer: "When the matching enrollment status flips to completed and the program's passing rules are satisfied. Edushade looks up the source entity, snapshots its title and description, captures the learner's grade and completion date, generates a number and verification code, and writes the issued-at timestamp."
    - question: "Can I issue a certificate manually?"
      answer: "Yes. The issue endpoint accepts an enrollment ID and the entity reference and creates a certificate record outside the auto-issue path — useful for makeup grading, special recognition, or programs where completion is decided by a human reviewer."
    - question: "What if a learner has already received a certificate?"
      answer: "Edushade checks for an existing certificate on the same enrollment before issuing. Duplicates are rejected with a clear error, so an enrollment cannot end up with two valid certificates by accident."
    - question: "What is the difference between the certificate number and the verification code?"
      answer: "The certificate number is the public reference — what a learner cites on their CV or shares in a profile. The verification code is the proof — a separate, harder-to-guess value that a third party uses to confirm the certificate is genuine. Two values, two purposes."
    - question: "What does the certificate record actually store?"
      answer: "Entity type and ID, optional cohort ID, learner ID and enrollment ID, title, description, status, certificate number, verification code, issued-at, expires-at and revoked-at timestamps, the snapshot grade (decimal precision), the completion date, total time spent, an optional template ID, and full audit fields (created-by, updated-by, soft delete)."
    - question: "Can a certificate expire?"
      answer: "Yes. Set expires_at on the record (or leave it null for evergreen credentials). The status moves to expired when the expiry date passes, and the record stays on file with the timestamp captured."
    - question: "Can a certificate be revoked?"
      answer: "Yes. Revocation flips the status to revoked and stamps revoked_at on the record. The record itself is not deleted — soft delete keeps the history, and the verification code stops returning a valid certificate from the public lookup."
    - question: "How does verification work?"
      answer: "A third party uses the verification code to confirm a certificate is genuine. The lookup returns the issued status, the snapshot grade, the completion date and the issuing entity title — enough to validate without exposing the rest of the platform."
    - question: "Are cohort certificates different?"
      answer: "Slightly. The certificate still references the underlying course or learning path as the entity, but the cohort_id is stored alongside so the record knows which batch the learner finished with. The same course can run multiple cohorts and the certificates stay attributable to the right run."
    - question: "Does the grade come from the program or the snapshot?"
      answer: "The snapshot. At issue time the learner's current_grade on the enrollment is copied onto the certificate as a decimal value. Later changes to the grading scheme do not change historical certificates."
    - question: "What admin metrics exist for certificates?"
      answer: "The certificates analytics view surfaces total issued, pending (completions awaiting issuance), average days from completion to issue, and the issuance rate as a percentage. All four track period-over-period trends."
    - question: "What states does a certificate go through?"
      answer: "Four: draft, issued, revoked and expired. Issued is the canonical happy path. Draft is used for records prepared but not yet released. Revoked and expired carry their own timestamps so a credential's full history reads cleanly off the record."
