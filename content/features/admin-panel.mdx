faq:
  title: "Admin Panel FAQ"
  description: "How the admin shell, RBAC, programs, billing, communication, audit and settings come together inside Edushade."
  items:
    - question: "What does an admin see on /admin/dashboard?"
      answer: "A greeting with the user's name, a Quick Actions row, a System Metrics widget, a course analytics chart, recent notices and enrollments in the main column, and a learning-stats card plus a top-learners leaderboard in the right rail. Behind it sits an Analytics group with Course Analytics, Enrollment Trends, Course Performance and more dedicated screens."
    - question: "How is the sidebar organised?"
      answer: "Seven groups: Dashboard, Analytics, Resources, Sales, Booking, Communication and Platform. Resources covers programs and content. Sales covers enrollments and billing. Booking covers meeting plans, schedules and integrations. Platform groups people, the website, audit and settings. Each item is keyed to roles and permissions so users only see what they can use."
    - question: "How does role-based access work?"
      answer: "Two layers. The base sidebar config carries `requireRoles` and `requirePermissions` per item — admins see the full menu, instructors see a teaching-scoped subset, custom roles see exactly what you grant. The same permissions enforce on the server, so a hidden item is also a closed API. Per-user overrides on top of roles let you grant or deny one permission to one person without changing the role."
    - question: "Which programs are managed from the admin panel?"
      answer: "Learning Paths, Courses, Quizzes, Cohorts, Events, Bundles, plus Categories and Tags as classification, and Products with their own taxonomies for the e-commerce side. Each ships its own list view, detail tabs and create / edit / publish / archive flows, and they all share a single Content Bank for files."
    - question: "What does the billing surface cover?"
      answer: "Per-entity pricing (course, cohort, learning path, bundle, event, meeting plan, product), EMI plans with active-installment views, scholarships and applications, subscriptions with active and failed views, offers with usage analytics, plus full transactions including pending, refunds and create / edit. The same admin role manages every billing screen."
    - question: "How is communication handled?"
      answer: "Three sub-areas under Communication. Discussion is the threaded discussion surface with audience-aware filtering. Noticeboard is the announcement system with priority, expiry, scheduling, pinning and read tracking. Notification covers transactional alerts with templates, segments, campaigns, settings and a full audit log."
    - question: "What does the audit surface ship?"
      answer: "Two screens under Platform → Audit. Audit Logs records every privileged change — role assignments, permission grants, settings updates and similar — with the actor, the entity and the before-after. Activity Logs records authenticated user activity for security review. Both are gated behind the audit.read permission, so only the right roles see them."
    - question: "What lives under Settings?"
      answer: "Four areas. Control Panel handles general org configuration. Holidays gives the platform a calendar of non-working days that other features can read. Integrations covers third-party connections. Payment Gateways covers the gateways used by the billing service. All four are admin-only, gated behind admin.settings or matching permissions."
    - question: "Is the website builder part of the admin panel?"
      answer: "Yes. The Website item under Platform links into the website-builder admin — pages, blocks, posts, forms, menus, settings — without leaving the same shell. The public site, blog and booking pages all flow from there, and they share the localization, SEO and theme configuration that the rest of the platform reads."
    - question: "Can I run bulk operations?"
      answer: "Yes. Every admin data table — users, courses, enrollments, transactions, reviews and beyond — exposes bulk-select with CSV export and bulk-delete by default. Bulk enrollment, bulk certificate issuing and bulk invitation flows live alongside the per-row actions, scoped by the same permissions."
    - question: "Is the admin panel multi-tenant?"
      answer: "Yes. Every entity carries a tenant_id and the entire panel is scoped at the database level — admins on one tenant cannot see data from another tenant. Tenant-level configuration (billing, notifications, site settings, roles, users, analytics tracking) is owned by each tenant's admins. Cross-tenant operations live behind a system token, not the user-facing panel."
    - question: "Is everything localized?"
      answer: "Yes. Sidebar labels, page headers, table columns, button text, modal copy and toasts all pull from the active locale. Content owned by the platform — courses, posts, forms, menus, blocks, SEO — uses the same `LocalizedField` shape so admins translate and the panel renders in every enabled language."
    - question: "How does the admin panel relate to the educator and learner sides?"
      answer: "The admin panel is the shared shell for staff. Educators sign in to the same /admin and see a teaching-scoped subset of the menu (no billing, no audit, no system settings, no website builder). Learners sign in to a separate panel at /dashboard with their own sidebar and their own personal dashboard. Each surface is a real product, wired into the same backend."
